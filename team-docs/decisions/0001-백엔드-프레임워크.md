## 맥락
문서 간 불일치 발견:
- `README.md` 기술스택 표: 백엔드 **NestJS**
- `design-docs/설계문서.md` 1장: **Next.js (App Router, Route Handlers)** — 풀스택 단일 코드베이스

두 문서가 서로 다른 걸 말하고 있어 무엇을 따라야 할지 모호하다. 코드 작성 전에 하나로 확정해야 한다.

## 결정
**아직 미확정.** 아래 트레이드오프를 보고 정한다.

| | Next.js Route Handlers | NestJS |
|---|---|---|
| 코드베이스 | 프론트·백 단일 | 분리(apps/web + apps/api) |
| 1인 운영 부담 | 낮음 | 중간 |
| 구조화·확장성 | 중간 | 높음 (DI, 모듈) |
| 현재 레포 구조 | `apps/web`만 있으면 됨 | 지금의 모노레포(apps/api)와 맞음 |

## 근거
- 대상 규모가 "동시 100명 / 누적 50~100명"으로 작고 1인 풀스택 → **단순함이 우선**이라면 Next.js.
- 단, 현재 레포는 이미 `apps/api`(NestJS 전제) 모노레포로 세팅돼 있음 → 되돌리는 비용도 고려.

## 결과 / 영향
- 확정되면 **둘 중 지는 쪽 문서를 수정**해 일치시킨다 (README ↔ 설계문서).
- 모노레포 유지 여부도 함께 결정된다.
- **확정 전까지 `apps/api` 본격 구현 보류** 권장.
