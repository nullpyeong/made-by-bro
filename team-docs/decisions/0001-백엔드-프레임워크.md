# 0001. 백엔드 프레임워크 선택

| | |
|---|---|
| 상태 | ✅ 채택 |
| 날짜 | 2026-06-21 |
| 파트 | Engineering |
| 결정자 | 김지평 |

## 맥락
문서 간 불일치 발견:
- `README.md` 기술스택 표: 프론트 **React + Vite** / 백엔드 **NestJS**
- `design-docs/설계문서.md` 1장: **Next.js (App Router, Route Handlers)** — 풀스택 단일 코드베이스

두 문서가 서로 다른 걸 말하고 있어 무엇을 따라야 할지 모호했다. 코드 작성 전에 하나로 확정한다.

## 결정
**React(+Vite) 프론트 + NestJS 백엔드의 분리형 모노레포로 확정한다.**
- `apps/web` — React + Vite + TypeScript + Tailwind (SPA)
- `apps/api` — NestJS + TypeScript (모듈/DI 기반 REST API)
- `packages/shared` — 프론트·백 공유 DTO/타입

| | Next.js Route Handlers (기각) | **React + NestJS (채택)** |
|---|---|---|
| 코드베이스 | 프론트·백 단일 | 분리(apps/web + apps/api) |
| 1인 운영 부담 | 낮음 | 중간 |
| 구조화·확장성 | 중간 | 높음 (DI, 모듈) |
| 현재 레포 구조 | `apps/web`만 | 지금의 모노레포(apps/api)와 일치 |

## 근거
- 현재 레포가 이미 `apps/api`/`apps/web` 모노레포로 세팅돼 있어 **되돌리는 비용 없이 그대로 진행** 가능.
- 프론트·백 책임 분리가 명확해 어드민·결제·영상 서명 등 백엔드 로직이 커져도 구조가 버틴다 (NestJS 모듈/DI).
- 단순함(Next.js 단일 코드베이스)의 이점보다, 학원 비즈니스 특성상 백엔드 도메인(결제·수강권·영상보호)이 빠르게 무거워질 것을 우선했다.

## 결과 / 영향
- **패자 문서(`design-docs/설계문서.md`)를 React+NestJS 기준으로 수정**해 README와 일치시킨다 (이 ADR과 같은 커밋).
  - 1장 기술 스택: Front(React+Vite) / Backend(NestJS) 분리
  - 2장 아키텍처: Vercel(React SPA) + Railway 등(NestJS API)
  - 5장 폴더 구조: `apps/web`(React) + `apps/api`(NestJS 모듈) 모노레포 기준
- 배포: 프론트 Vercel, 백엔드 Railway/Render(서버 인스턴스). 기존 "Vercel 단일" 전제는 무효.
- `apps/api` 본격 구현 착수 가능.
