# 0010. 웹 React 포팅 — CSS 전략 (기존 app.css 재사용)

## 맥락
ADR 0001로 `apps/web` = React+Vite+TS가 확정됐고, 이제 `docs/`의 정적 시안(10개 HTML + `app.css` 1514줄 + `tokens.css` + `app.js` 1646줄)을 실제 React 앱으로 옮긴다.
- `SETUP.md` 2절은 프론트 CSS로 **Tailwind**(토큰 매핑)를 제안한다.
- 그러나 `app.css`는 수십 차례 다듬어 완성도 높은 디자인 시스템(히어로·벤토·요금제·플레이어·차트·다크모드·CJK 줄바꿈 등)이 이미 토큰 기반으로 구현돼 있다.
- 코드 작성 전에 "기존 CSS를 Tailwind로 다시 쓸지 / 그대로 재사용할지"를 정해야 한다.

## 결정
**`tokens.css` + `app.css`를 그대로 전역 스타일로 import 하고, JSX는 동일 className(`.hero`·`.gnb`·`.kpi`…)을 사용한다.** Tailwind 전면 재작성은 하지 않는다. (필요 시 *새* 컴포넌트에 한해 Tailwind를 선택적으로 추가 가능 — SETUP.md의 Tailwind 라인은 그 용도로 남긴다.)

| | Tailwind 전면 재작성 (SETUP 원안) | **app.css 재사용 (채택안)** |
|---|---|---|
| 디자인 보존 | 1514줄 재작성 → 폴리시 드리프트 위험 | 100% 그대로 보존 |
| 작업량 | 큼 (전 컴포넌트 유틸 변환) | 작음 (HTML→JSX + 클래스 유지) |
| 스코프 | 유틸 단위 | 전역(기존 구조 그대로) |
| 진짜 작업의 초점 | CSS 재작성에 분산 | `app.js` 로직의 React화에 집중 |

## 근거
- 마이그레이션의 진짜 난이도는 CSS가 아니라 **`app.js`의 상태/로직(장바구니·진도모델·강의실 엔진·대시보드)을 React로 옮기는 것**이다. CSS를 재작성하면 그 본질 작업에서 자원이 새고, 검증된 비주얼이 깨질 위험만 커진다.
- `tokens.css`가 디자인 SSOT이고 `app.css`는 그 위에 빌드돼 있어, import만으로 `docs/` 시안과 픽셀 동일하게 재현된다. (Home 포팅 후 헤드리스 렌더로 동일성 확인 완료.)
- Tailwind는 포기가 아니라 **유보**다. 새 화면/컴포넌트에서 점진 도입할 여지를 남긴다.

## 결과 / 영향
- `apps/web/src/styles/`에 `tokens.css`·`app.css` 복사본(현재는 `docs/`에서 복사). → **이후 `packages/shared`로 끌어올려 `docs`/`web`이 한 소스를 공유**하는 게 다음 정리 과제(현재는 중복).
- 정적 화면(home 등)은 "HTML→JSX + `app.js` 동작을 effect로 연결"하는 방식으로 빠르게 포팅. 동적 화면(player·cart·checkout·mypage)은 `app.js` 로직을 **React 상태로 정식 재구성**.
- `SETUP.md` 2절의 "Tailwind 전면" 전제는 무효화 → 새 컴포넌트 한정 옵션으로 격하(이 ADR 채택 시 SETUP.md 갱신).
- 리스크: 전역 CSS라 컴포넌트 스코프 격리가 없음(기존 구조와 동일하므로 신규 리스크는 아님).
