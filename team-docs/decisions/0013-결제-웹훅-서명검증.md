## 맥락
0003에서 PG를 토스페이먼츠로 정했고, 결제 확정은 **웹훅(서버-서버 콜백)**으로 통지된다. 이 웹훅은 사용자 JWT가 없는 외부 호출이라 전역 `JwtAuthGuard`(ADR 0011)를 우회해야 한다. 그런데 그냥 열어두면 **누구나 "결제 완료" 웹훅을 위조**해 수강권을 공짜로 얻거나 추천 전환(CAC)을 오염시킬 수 있다. → 웹훅 출처를 암호학적으로 검증해야 한다.

또한 0006의 "유료 전환 → 추천인 CAC 기록"을 **실제 결제에 자동 연결**하려면, 지금까지 수동(`POST /rewards/conversion`)이던 전환을 웹훅이 직접 트리거하는 경로가 필요하다.

## 결정 (제안 — 확정 대기)
**원문 바디 HMAC-SHA256 서명 검증 + fail-closed.**

- **`WebhookSignatureGuard`**: `req.rawBody`(원문 Buffer)에 `HMAC-SHA256(TOSS_WEBHOOK_SECRET)`를 계산해 서명 헤더와 **timing-safe 비교**. 불일치/누락이면 401.
- **fail-closed**: `TOSS_WEBHOOK_SECRET` 미설정이면 **모든 웹훅 거부**(실수로 열리는 사고 방지).
- **rawBody 보존**: `NestFactory.create(AppModule, { rawBody: true })` — 서명은 파싱 전 원문 기준이어야 위조가 막힌다.
- **자동 전환 연결**: 서명 통과 → `paymentKey`(=`payments.pg_tid`)로 결제 건 조회 → 완료 상태면 `status=paid` 전이(멱등) → `RewardsService.recordConversion` 호출로 추천 전환·CAC 자동 기록.
- **엔드포인트**: `POST /api/payments/webhook/toss` (`@Public` + `WebhookSignatureGuard`).

## 근거
- **원문 HMAC** — 바디 1바이트만 바뀌어도 서명 깨짐. 표준적 웹훅 위조 방지.
- **fail-closed** — 시크릿 누락 시 "검증 없이 통과"가 아니라 "전부 거부". 보안 기본값.
- **멱등(updateMany where status≠paid)** — 토스가 웹훅을 재전송해도 전환·보상 1회만.

## ⚠️ 유보 (사람 확정 + 연동 시 문서 대조 필요)
1. **Toss의 실제 서명 사양** — 헤더명·인코딩·서명 대상(전체 바디 vs 특정 필드)을 **공식 문서로 확정**해야 한다. 현재 구현은 env로 바꿀 수 있게 열어둠:
   - `TOSS_WEBHOOK_SIG_HEADER` (기본 `tosspayments-webhook-signature`)
   - `TOSS_WEBHOOK_SIG_ENCODING` (기본 `base64` | `hex`)
   - 토스가 HMAC 서명을 제공하지 않는 경우의 **대안**: 웹훅 수신 후 `paymentKey`로 토스 결제조회 API를 **secret key로 재조회**해 금액·상태를 검증(re-query confirm). 이 경우 가드 대신 서비스에서 재조회 검증.
2. **완료 상태값 집합** — 현재 `DONE/PAID/COMPLETED`로 가정. 토스 실제 상태값으로 좁혀야 함.
3. **결제 생성(checkout) 플로우 부재** — 지금 웹훅은 *사전 생성된 결제 row*를 확정하는 경로만. 결제 시작(주문 생성→`pg_tid` 발급)은 후속.

## 결과 / 영향
- **신규**: `payments` 모듈(`PaymentsController`/`PaymentsService`/`WebhookSignatureGuard`).
- **환경변수**: `TOSS_WEBHOOK_SECRET`(필수, 미설정 시 웹훅 전부 거부), `TOSS_WEBHOOK_SIG_HEADER`, `TOSS_WEBHOOK_SIG_ENCODING`.
- **검증**: temp(OneDrive 밖) 빌드 + 실서버 e2e — 유효 서명만 통과, 위조/시크릿 누락 거부, 결제 paid 전이 + 추천 converted 자동 연결, 재전송 멱등.
