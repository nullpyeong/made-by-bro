# 0008. 커머스 모델 — 단건 + 멤버십(구독) 병행 구조

| | |
|---|---|
| 상태 | ✅ 채택 (구조 한정) |
| 날짜 | 2026-06-21 |
| 파트 | Engineering / Product |
| 결정자 | 김지평 |

## 맥락
기존 스키마(`design-docs/schema.sql`)는 **단건 판매 전용**이었다.
- 결제: `payments.course_id NOT NULL` (반드시 특정 코스 1개)
- 접근 권한: `enrollments`(UNIQUE(user_id, course_id))로 코스 1개씩 부여

그런데 0005(요금제 가격)에서 **멤버십 월/연 구독**을 가격 정책으로 잡았고, 0006(60명 시딩)은 "3개월 무료 접근"이라는 **구독성 접근**을 전제한다. 단건 모델만으로는:
- "구독 active → 전체 강의 접근"이라는 접근 판정 경로가 없다.
- 토스 빌링키 기반 **자동결제(반복 청구)**를 담을 곳이 없다.
- 시딩 60명의 "무료 3개월 접근"을 표현할 구조가 없다.

가격(0005)은 아직 🟡 잠정이지만, **구조(스키마)는 지금 박아야** 백엔드(`apps/api`)·결제·시딩 작업이 같은 토대 위에서 시작된다.

## 결정
**단건을 유지한 채, 그 옆에 멤버십(구독) 구조를 얹는다.** 강의 접근 판정을 "단건 OR 구독" 두 경로를 합쳐서 한다.

추가/변경한 구조:
- **`plans`** — 멤버십 요금제 정의(월/연). 단건 가격은 그대로 `courses.price` 사용.
- **`subscriptions`** — 구독 상태의 원천. 토스 `billing_key`(자동결제), `current_period_end`(접근 만료), `source`(`paid`/`seed`=0006 시딩/`comp`=무상), `cancel_at_period_end`(기간말 해지 예약). **1유저 동시 활성 구독 1개**(부분 유니크).
- **`payments` 확장** — `course_id`를 nullable로 바꾸고 `subscription_id` 추가. `CHECK (course_id IS NOT NULL OR subscription_id IS NOT NULL)`로 "단건 또는 구독, 둘 중 하나"를 강제. 단건/구독 결제를 한 원장에 통합.
- **접근 판정(앱 레벨)**:
  `canAccess(user, course) = 활성 enrollment(단건) OR 활성 subscription(멤버십=전체 published 코스)`

## 근거
- **단건을 안 깬다** — 기존 enrollment/refund/receipt 로직이 그대로 살아있다. 멤버십은 *추가* 경로일 뿐.
- **payments 통합 원장** — 단건·구독 결제를 한 테이블에서 환불·영수증·정산까지 동일 흐름으로 처리. 별도 invoice 테이블을 안 만들어 단순.
- **시딩을 구조로 흡수** — 0006의 무료 3개월을 `subscription(source='seed', billing_key NULL, period_end=+3개월)`로 자연스럽게 표현. 별도 "시드 계정" 플래그 불필요.
- **버린 대안**:
  - *멤버십 전용 access 테이블 신설* → enrollment와 판정 경로가 둘로 갈려 복잡. 기각.
  - *구독 결제용 invoices 테이블 분리* → 100명 규모에 과설계. payments 재사용이 단순.
  - *멤버십 구조 자체를 보류(미정 ADR)* → 0006 시딩·결제 작업이 토대 없이 떠서 헛작업 위험. **그래서 구조는 지금 박는다.**

## 결과 / 영향
- **[Eng]** `apps/api` 결제/접근 모듈은 이 스키마 기준으로 구현:
  - 접근 판정 유틸 1개(단건 OR 구독)로 모든 강의 게이팅 통일.
  - 구독 갱신 배치: `current_period_end` 임박 active 구독을 `billing_key`로 재청구 → payments row 생성 + 주기 +1, 실패 시 `past_due`(유예 후 `expired`).
  - 만료 배치: `current_period_end < now()` → `expired`.
- **[0006 시딩]** 60명 = `source='seed'` 구독 일괄 생성으로 처리 가능. (추천코드·코호트·활성화 로깅은 여전히 별도 후속 — 이 ADR 범위 밖)
- **[얼리버드 잔여석(G4)]** 이 ADR 범위 아님. 별도 `offers` 테이블은 후속 결정/작업으로 남김.
- **리스크**: 단건 보유자가 멤버십까지 가입하면 *접근이 중복*되지만 권한상 문제는 없음(둘 다 허용). 환불 시 단건/구독 정책이 달라 분기 필요.
